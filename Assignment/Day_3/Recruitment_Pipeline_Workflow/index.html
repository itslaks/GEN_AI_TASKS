LangGraph Architecture — Recruitment Pipeline

Production-grade StateGraph with 3 AI agents, 2 decision routers, error handling, retries, and idempotency. Each node operates on a shared typed state. LLM calls are isolated from deterministic logic.

State Schema
class RecruitmentState(TypedDict):
  job_id:              str
  job_description:     str
  resumes:             List[dict]
  screened_candidates: List[ScreenedCandidate]
  scheduling_requests: List[SchedulingRequest]
  interview_slots:     List[InterviewSlot]
  interview_notes:     List[InterviewNote]
  evaluations:         List[Evaluation]
  audit_log:           List[AuditEntry]
  clarification_needed: List[str]
  scheduling_retries:  dict
  error:               Optional[str]
Extension Notes
ATS Integration
Replace stub tools with Greenhouse/Lever API calls. Map candidate_id to ATS record IDs.

Email Notifications
Add SendGrid/SES in request_clarification_node and post-evaluation node.

Scoring Calibration
Store rubric weights per job_id. Add feedback loop node to adjust thresholds.
Tool Boundaries
■ LLM (Groq) — Resume parsing, slot generation, rubric evaluation
■ Deterministic — Score extraction, JSON parsing, deduplication
■ Stub Tools — calendar_api_check_availability, calendar_api_book_slot
■ Idempotency — dedupe_by_candidate_id (MD5 hash fallback)
Error & Retry Strategy
• LLM parse failure → deterministic fallback score
• Calendar conflict → max 2 retries with offset slots
• Low confidence (<60%) → human_review queue
• All events logged to audit_log with timestamps
• Candidate IDs deduplicated on every run
